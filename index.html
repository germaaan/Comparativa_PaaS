Uso de diferentes PaaS

Germán Martínez Maldonado

PaaS

En un PaaS solo nos tenemos que preocupar de gestionar nuestra aplicación y los datos del mismo, ya que del resto de aspectos se encarga el proveedor del servicio:

  • Integración
  • Base de datos
  • Sistema operativo
  • Virtualización
  • Servidores
  • Almacenamiento
  • Red

PaaS

Los PaaS son compatibles con diferentes pilas, marcos de aplicación o lenguajes de programación (buildpacks) que nos facilitan el despliegue de nuestras aplicaciones, además de darnos la posibilidad de añadir complementos que podremos usar externamente desde nuestra aplicación.

Por ejemplo, tanto Heroku como OpenShift nos permiten añadir como complemento una base de datos PostgreSQL sobre la que podríamos trabajar directamente con nuestra aplicación.

PaaS

Para desplegar nuestros aplicaciones los PaaS proveen al usuario de máquinas virtuales con tal fin.

Estas máquina pueden ser escaladas en función de las necesidades, dependerá de la plataforma y del tipo de aplicación la forma en que podamos escalar nuestras aplicaciones.

PaaS

En Heroku tenemos los dynos y principalmente podemos diferenciar dos tipos: web para los procesos que van a recibir tráfico HTTP y worker para los procesos que se van a ejecutar en segundo plano.

También podemos añadir funcionalidades extras mediante los addons.

PaaS

En OpenShift tenemos los gears y cartridges: los gears son los que contendrán nuestras aplicaciones y los cartridges tienen el objetivo de añadir funcionalidades concretas.

PaaS

Bluemix utiliza la plataforma open-source Cloud Foundry para proveernos el servicio de PaaS.

Heroku

Los dynos de Heroku se configuran mediante un archivo llamado Procfile. 


            PROCESO: COMANDO

            web: npm start

Heroku

Desde un terminal mediante el cliente de línea de comandos de Heroku podemos arrancar o modificar la configuración de los dynos.

Heroku

Podemos escalar horizontalmente añadiendo más dynos: 

heroku ps:scale web=2

Pero también podemos escalar verticalmente usando dynos más grandes: 

heroku ps:scale web=2:standard-2x

La escalabilidad de los dynos es independiente entre sí.

Heroku

OpenShift

En OpenShift las aplicaciones se crean independientemente y se ejecutan en los application gears. Además, es muy común que se les vaya añadiendo los complementos que necesiten mediante lo que se conoce como cartuchos.

Tambien desde terminal podemos arrancar o modificar la configuración de nuestra aplicación.

OpenShift

El escalado de la aplicaciones es automático y la propia plataforma puede añadir o eliminar gears en función de las necesidades que vaya detectando.

Sin embargo, para que esto sea posible la aplicación debe ser creada como una aplicación escalable.

OpenShift

Para crear una aplicación escalable deberemos indicar el número máximo y mínimo de gears de los que va a disponer la aplicación para su escalado. 

rhc cartridge scale CART -a APP --min MIN --max MAX

OpenShift

Si una aplicación no se crea como escalable, todo el tráfico es mandado a un único gear.

Si añadimos otros cartuchos (ej: base de datos), estos compartirán el mismo gear y los recursos del gear de nuestra aplicación.

OpenShift

Si una aplicación se crea como escalable, automáticamente se le añade un cartucho con HAProxy para balancear la carga.

Cuando el tráfico aumenta, HAProxy avisa a OpenShift, crea una copia de nuestra aplicación en otro gear (siempre que dispongamos de más).

Bluemix

Bluemix es una plataforma PaaS de IBM, pero que también nos proporciona servicios de IaaS dando la posibilidad de contratar por servicios de máquinas virtuales.

Bluemix

En lo que al ámbito PaaS se refiere, podemos desplegar nuestras aplicaciones directamente en la plataforma Cloud Foundry.

Esta plataforma nos ofrece también diferentes servicios como MySQL, MongoDB, PostgreSQL, Redis, RabbitMQ...

Bluemix

El escalado en Bluemix puede ser tanto automático como manual.

En el caso del modo automático, se puede activar simplemente añadiendo el plugin necesario y añadiendo un política de escalado a nuestra aplicación.

bx as policy-attach APP -p FILE

Bluemix

En función del tipo de aplicación es posible definir reglas de escalado para el almacenamiento dinámico, la memoria o el rendimiento.

No se detectan de forma automática conflictos entre políticas de escalado. Un conflicto podría hacer que se escale verticalmente un minuto y horizontalmente al siguiente.

Bluemix

Campos que se pueden usar para la creación de políticas:

  • Recuento máximo de instancias permitidas
  • Duración de la infracción
  • Periodo de enfriamiento del escalado vertical/horizontal
  • Hora de inición/finalización
  • Repetición activada

Comparación

  • En los tres PaaS se puede escalar tanto horizontal como verticalmente, pero Heroku es el único en el que hay que hacerlo de forma manual.
  • Aunque todos soportan de forma nativa los lenguajes de programación más comunes, Heroku soportan además lenguajes más especificos como Scala o Clojure.

Comparación

  • Heroku es el PaaS que más frameworks soporta, seguido de cerca por OpenShift. Bluemix solo soporta Rails y Sinatra, siendo una desventaja ya que ambos son muy similares.
  • Bluemix es el PaaS que más services provee de forma nativa (49) que van desde acceso móvil a gestión de base de datos o traducción automática. Heroku solo provee gestiones de base de datos y Openshift solo añade a eso servicios de integración continua.

Comparación

  • Sin embargo, Heroku es el que dispone de un mayor número de complementos (179) también con muchas finalidades diversas como integración continua o gestión de base de datos o gestión de información sobre las aplicaciones. Los complementos de OpenShift y Bluemix están más bien enfocados a la gestión de base de datos únicamente.
